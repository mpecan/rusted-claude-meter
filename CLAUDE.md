Rusted Claude Meter is a Tauri v2 tray app (macOS + Linux) showing Claude plan usage — a port of the SwiftUI ClaudeMeter.

Layout: `crates/meter-core` = pure domain (no I/O); `crates/meter-api` = claude.ai HTTP client + response mapping; `crates/meter-render` = pure tray-icon rendering (`IconState` → SVG → RGBA, no platform code); `src-tauri` = app shell (tray, scheduler, storage, notifications); `src/` = vanilla TS frontend.

Run `just check` before considering any change done — it is what CI enforces: `cargo fmt --check`, `cargo clippy --workspace --all-targets -- -D warnings` (pedantic + nursery on; `unwrap_used`/`expect_used`/`panic`/`todo` are deny), `cargo test --workspace`, the 500/700-line file-size gate, `cargo deny check` (licenses/advisories/bans/sources, `deny.toml`), `cargo dupes check` (structural duplication ceiling), `cargo llvm-cov` (coverage floor), and the frontend (`tsc --noEmit` + `npm run test`). Test modules may `#![allow(clippy::unwrap_used)]`. CI runs on Linux only (`.github/workflows/ci.yml`) — the macOS test/clippy leg was dropped on purpose (the app is developed on macOS, and macOS runners cost 10x the minutes), so the `#[cfg(target_os = "macos")]` surface (popover, tray click handling) is held to the same `-D warnings`/test gate *only* by running `just check` locally on a Mac; a Linux-only check (CI or a Linux container) never compiles it. One CI-only check has no local `just` recipe: actionlint over `.github/workflows/*.yml`.

Quality gates in `justfile`/`ci.yml` are ratchets: the coverage floor (`coverage_min_*`) only ever goes up, the duplication ceiling (`dupes_max_*`) only ever goes down. Never loosen either to make a change pass — add tests or remove the duplication instead. `deny.toml`'s `[advisories] ignore` list holds only pre-existing "unmaintained" transitive notices with no safe upgrade (tauri/gtk-rs tree); don't add a new entry there without checking whether the flagged crate is a real, fixable dependency of ours first.

The frontend must be built once (`npm run build`) before any cargo command touching `src-tauri` — `tauri-build` requires `dist/` to exist.

Model-scoped limits contract: the API's `limits` array is the source of truth for per-model caps; entries are keyed by `scope.model.display_name` (API-supplied, e.g. "Sonnet", "Fable"), headline kinds (`five_hour`, `seven_day`) are excluded from the scoped pass, and incomplete entries are skipped, never errors. Flat fields are the headline fallback. Fixture: `crates/meter-api/tests/fixtures/usage_response.json`.

Pace-tracking contract (issue #16): pace ratio, `PaceBand`, projections and the hybrid `pace_signal` live in `meter_core::pacing`/`snapshot` and MUST stay numerically identical to `src/pacing.ts` (both file headers promise this — change them together). Ratio = `min(utilization,100) / expected%`, where expected is the elapsed fraction of the *pacing span* capped at 1.0; `None` under `MIN_ELAPSED_FRACTION` (5%) elapsed, but the limit-hit projection bypasses that grace while honouring the 5%-utilization floor. `pace_tracking_enabled` (default true) is the master switch — when off the popover computes no pace at all (ratio/expected/projection all null, quota-first only) and the tray folds it into the effective pace-first flag (`pace_tracking_enabled && pace_first_display`), so no separate field threads through the tray; `weekly_pace_days` (5–7, clamped) and `pace_first_display` are the two configurable sub-settings under it. `pace_signal` is headline-only (session + weekly, never scoped) and hybrid: overuse on either window, underuse only on weekly. In pace-first display the tray icon's primary metric always becomes a ratio (`pace_signal.ratio ?? session.pace_ratio ?? weekly.pace_ratio`, its band drives the colour), but the flame/snowflake badge and the "Used X% vs Y%" tooltip/pace-line are gated on the hybrid signal being present; the popover's per-card pace ratio/projection is always computed regardless of the toggle.

Linux tray reality: StatusNotifierItem gives no click events or tooltips — the tray menu is the primary Linux surface. The macOS popover is a native `NSPopover` hosting the webview (via the macOS-only `tauri-plugin-nspopover`, git dep pinned in `src-tauri/Cargo.toml` and allow-listed in `deny.toml`); it's `#[cfg(target_os = "macos")]`-gated so Linux still builds. Settings is a separate window; the popover renders one of two layouts (`popover_layout`: rows/cards) with status colours keyed to the user's warning/critical thresholds.
